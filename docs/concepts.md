# 💡 Fumika Engine Architecture Guide

본 문서는 `fumika` 엔진의 내부 설계 철학, 아키텍처 구조 및 핵심 메커니즘을 상세히 기술합니다.  
엔진의 작동 원리를 깊이 있게 이해함으로써 최적화된 게임 로직을 설계하고 시스템을 효과적으로 확장하실 수 있도록 안내해 드립니다.  

---

## 🏗️ 1. 핵심 아키텍처: 모듈형 MVC (Overview) <a id="overview"></a>

`fumika`는 웹 환경에서의 성능과 유지보수성을 극대화하기 위해 **단방향 데이터 흐름을 가진 모듈형 MVC 구조**를 채택하였습니다.  
이는 복잡한 시나리오 연출 과정에서도 상태의 일관성을 유지하기 위한 핵심적인 설계적 장치입니다.  

### **구성 요소별 역할 정의**
*   **Model (Variables)**: 모든 상태 데이터의 단일 진실 공급원(SSOT)입니다.  세이브 데이터로 직렬화될 수 있는 순수 객체(Plain Object) 형태를 유지해야 합니다.  
*   **Controller (Novel/Modules)**: 시나리오 명령어를 해석하여 Model을 조작하거나 렌더러에 시각적 명령을 전달하는 제어 계층입니다.  
*   **View (Renderer/Canvas)**: 상태 변화를 감지하고 Canvas API를 통해 고성능 렌더링을 수행하는 표현 계층입니다.  

이러한 분리 구조는 개발자가 비즈니스 로직(시나리오)과 표현 계층(렌더링)을 독립적으로 확장할 수 있게 합니다.  
특히 DOM 기반의 레이아웃 계산 부하를 우회하여 대규모 오브젝트를 안정적으로 처리할 수 있도록 돕습니다.

## 🎬 2. 실행 컨텍스트로서의 장면 시스템 (Scenes) <a id="scenes"></a>

`fumika`에서 씬(Scene)은 단순한 시나리오의 묶음이 아닌, **독립적인 실행 컨텍스트(Execution Context)**로 기능합니다.  
각 씬은 고유의 로컬 상태와 자원 관리 범위를 엄격히 가집니다.  

### **자원 관리 파이프라인**
엔진은 새로운 씬으로 진입할 때마다 다음과 같은 자원 관리 절차를 수행합니다:
1.  **자원 소거 (Cleanup)**: 이전 씬에서 생성된 모든 시각적 객체를 명시적으로 해제하여 GPU 메모리 누수를 방지합니다.  
2.  **상태 초기화 (Initialization)**: 공통 설정(`defineInitial`) 및 씬 전용 `initial` 데이터를 바탕으로 시스템 모듈의 초기 환경을 재설정합니다.  

씬의 `initial` 설정은 명령어가 실행되기 전, 모듈의 기본 성격을 규정하는 메타데이터를 주입하는 통로로서의 역할을 수행합니다.

## 🧪 3. 데이터 영속성을 위한 변수 시스템 (Variables) <a id="variables"></a>

변수 시스템은 게임 데이터의 **영속성(Persistence)**과 완벽한 세이브/로드 기능을 최우선으로 고려하여 설계되었습니다.  

### **상태 직렬화 정책**
`fumika` 엔진은 사용자의 진행 상태를 완벽하게 복원하기 위해, 엔진 내부에 존재하는 모든 상태 데이터를 직렬화하여 저장합니다.  
직렬화 대상에는 다음 데이터들이 포함됩니다:

*   **전역 변수 스냅샷**: `Novel` 인스턴스에 유지되는 전체 전역 변수 목록입니다.  
*   **지역 변수 스냅샷**: 현재 실행 중인 씬의 로컬 변수 목록입니다.  `_` 접두사를 통해 스코프가 격리되지만 세이브 데이터에는 안전하게 포함됩니다.  
*   **모듈 상태 (States)**: 대사 출력 상태, 오디오 트랙 정보 등 각 모듈의 내부 상태 데이터입니다.  
*   **렌더러 상태 (Renderer State)**: 카메라, 배경, 캐릭터 레이어의 시각적 구성 정보입니다.  

모든 상태를 순수 객체(Plain Object)로 관리함으로써, 엔진은 어느 시점에서든 현재 상태를 데이터화하여 보존할 수 있는 강력한 영속성을 확보합니다.

## ⚡ 4. 런타임 지능을 부여하는 Resolvable 패턴 <a id="resolvable"></a>

`Resolvable`은 정적인 속성값에 실시간 판단 로직을 결합하는 디자인 패턴입니다.  
이는 명령어의 속성이 고정된 값뿐만 아니라 함수를 수용할 수 있게 하여 연출의 유연성을 극대화합니다.  

### **작동 원리 및 제약 조건**
엔진은 명령어 실행 직전에 속성을 평가합니다.  
값이 함수일 경우 현재의 상태(`variables`)를 인자로 전달하여 즉시 실행하고, 그 결과값을 최종 속성값으로 채택합니다.  

렌더링 프레임의 일관성을 유지하기 위해 모든 Resolvable 함수는 **동기적(Synchronous) 실행**을 원칙으로 합니다.  
이는 시나리오 내부의 조건부 분기 코드를 획기적으로 간소화하는 데 기여합니다.

## ⚓ 5. 데이터 가공을 위한 파이프라인 훅 (Hooks) <a id="hooks"></a>

`fumika`의 훅 시스템은 단순한 이벤트 리스너를 넘어, 데이터를 가공하고 흐름을 제어하는 **인터셉터 파이프라인**으로서 작동합니다.  

### **릴레이(Relay) 데이터 흐름**
여러 개의 훅이 등록된 경우, 엔진은 이전 훅의 결과값이 다음 훅의 입력값으로 전달되는 릴레이 방식을 따릅니다.  
이를 통해 다국어 번역, 데이터 트래킹, 특정 상황에서의 진행 차단 등 고도화된 시스템 로직을 엔진 코어의 수정 없이 구현할 수 있습니다.  

---

## 🛠️ 6. 객체 추적 및 자원 회수 시스템 (Track & Clear) <a id="debug-mode"></a>

Canvas 기반 렌더링에서 자원 관리의 실수는 성능 저하의 직접적인 원인이 됩니다.  
`fumika`는 이를 체계적으로 관리하기 위해 **명시적 객체 추적 시스템**을 제공합니다.  

### **Renderer.track() 및 디버깅**
모듈에서 생성된 모든 시각적 객체는 엔진의 추적 목록에 등록되어야 합니다.  
엔진은 이 목록을 관리하며, 씬 전환 시 등록된 모든 객체의 소멸자를 호출하여 자원을 회수합니다.  
`novel.debugMode`를 활성화할 경우 추적 중인 객체들의 물리적 영역을 시각화하여 레이아웃 검증을 지원합니다.
