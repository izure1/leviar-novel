# 🧩 Custom Module Guide

`fumika` 엔진의 모든 기능은 독립적인 **모듈(Module)** 단위로 설계되었습니다. 본 가이드는 엔진의 코어 시스템과 유기적으로 상호작용하는 커스텀 모듈을 제작하기 위한 설계 철학과 기술 명세를 제공합니다.

---

## 🚀 빠른 시작 (Quick Start)

가장 직관적인 예제인 '알림(Notice)' 모듈을 통해 구조를 파악해 봅니다.

```ts
import { define } from 'fumika'

// 1. 모듈 정의 (상태 및 커맨드 타입 선언)
export const noticeModule = define<{ message: string }, { currentMessage: string }>({
  currentMessage: '' // 초기 상태 스키마
})

// 2. 커맨드 핸들러 (비즈니스 로직)
noticeModule.defineCommand(function* (cmd, ctx, state, setState) {
  setState({ currentMessage: cmd.message }) // 상태 업데이트
  
  const dur = ctx.renderer.dur(2000)
  ctx.execute({ type: 'control', action: 'disable', duration: dur })
  
  yield false // 연출이 완료될 때까지 대기
  return true // 커맨드 종료 및 다음 흐름 진행
})

// 3. 뷰 렌더러 (UI 구현)
noticeModule.defineView((data, ctx) => {
  return {
    show: (dur) => { /* 진입 연출 */ },
    hide: (dur) => { /* 퇴장 연출 */ },
    onUpdate: (d) => { /* 데이터 변경 시 렌더링 갱신 */ },
    onCleanup: (d) => { /* 모듈 제거 또는 씬 전환 시 정리 */ },
  }
})
```

## 💎 개발 프로세스 요약

| 단계 | 작업 내용 | 핵심 API |
| :--- | :--- | :--- |
| **1. Model 정의** | 모듈이 관리할 데이터 구조 및 초기값 설계 | `define(schema)` |
| **2. Controller 구현** | 시나리오 명령 실행 시의 로직과 흐름 제어 | `defineCommand()` |
| **3. View 구현** | 데이터를 기반으로 실제 화면을 구성하는 렌더러 | `defineView()` |

---

## 🔄 상세 구현 가이드

### 1. 상태 관리 (Model)

`define`으로 선언된 데이터는 모듈의 유일한 상태 소스(Single Source of Truth)가 됩니다.

*   **상태 변경 규칙**: 반드시 **`setState`**를 통해서만 변경해야 합니다. 이는 반응형 엔진이 상태 변화를 감지하여 뷰를 동기화하는 유일한 통로입니다.
*   **배치 업데이트**: 여러 번의 `setState` 호출은 엔진 내부에서 최적화되어, 한 번의 렌더링 사이클로 병합 처리됩니다.

### 2. 커맨드 핸들러 (Controller)

시나리오의 명령을 처리하는 제너레이터 함수입니다.

*   **인자 활용**:
    *   `cmd`: 변수가 모두 처리된(Resolved) 최종 데이터입니다.
    *   `ctx`: 엔진의 모든 API(Renderer, Scene, UI)에 접근하는 통합 인터페이스입니다.
*   **흐름 제어**:
    *   `yield false`: 사용자 입력이나 애니메이션 완료를 기다리며 실행을 일시 중단합니다.
    *   `return true`: 현재 명령의 수행을 완료하고 다음 명령으로 제어권을 넘깁니다.

### 3. UI 렌더러 (View)

엔진의 라이프사이클과 가시성 시스템에 대응하는 객체를 반환합니다.

#### 💡 가시성 및 생명주기 메서드
| 메서드 | 호출 시점 | 구현 가이드라인 |
| :--- | :--- | :--- |
| **`show`** | `ui` 커맨드 또는 **`novel.showUI()`** 호출 시 | UI를 다시 표시하는 **진입 연출**을 포함하십시오. 시스템의 '인터페이스 표시' 요청 시에도 호출됩니다. |
| **`hide`** | `ui` 커맨드, 레이어 간섭, 또는 **`novel.hideUI()`** 호출 시 | UI를 숨기는 **퇴장 연출**을 구현하십시오. 우클릭 등을 통한 '인터페이스 숨기기' 요청 시에도 호출됩니다. |
| **`onUpdate`** | 상태 데이터 변경 시 | 변경된 데이터를 UI 요소에 즉시 투영하십시오. |
| **`onCleanup`** | 모듈 제거 또는 씬 전환 시 | 이벤트 리스너 해제 등 모든 자원을 정리하십시오. |

#### 💡 레이어 관리 및 자동 제어 (uiGroup & hideGroups)

엔진은 각 모듈이 선언한 **`uiGroup`** 식별자를 기반으로 UI 가시성을 지능적으로 관리합니다. 이 시스템이 왜 필요하고 어떻게 작동하는지 **단계별 예시**를 통해 알아보겠습니다.

##### 🛠️ 실전 시나리오 예시

**상황**: 평소에는 대화창이 떠 있다가, 사용자의 이름을 묻는 **입력창**이 나타날 때는 대화창이 깔끔하게 사라져야 합니다. 입력이 끝나면 다시 대화창이 나타나야 하죠.

**Step 1: 모듈에 '이름(ID)' 부여하기**
먼저, 제어의 대상이 될 모듈에게 식별 가능한 이름을 붙여줍니다. 
예를 들어, 대화창 모듈은 스스로를 **`'main-ui'`**라고 명명합니다.

```ts
// dialogue 모듈의 defineView 내부
return {
  uiGroup: 'main-ui', // "내 이름은 이제부터 'main-ui'입니다."
  show: (dur) => { /* 나타나는 애니메이션 */ },
  hide: (dur) => { /* 사라지는 애니메이션 */ },
}
```

**Step 2: '방해 금지 규칙' 설정하기**
이제 입력창 모듈에게 "네가 나타날 때는 특정 이름을 가진 UI들을 숨겨라"라고 알려줍니다.
예를 들어, 입력창 모듈의 **`hideGroups`**에 위에서 정한 **`'main-ui'`**를 적어줍니다.

```ts
// input 모듈의 defineView 내부
return {
  uiGroup: 'input-layer', 
  // "내가 화면에 나올 때는 'main-ui' 그룹에 속한 녀석들을 모두 hide() 시켜주세요."
  hideGroups: ['main-ui'], 
  show: (dur) => { /* 입력창 나타남 */ },
  hide: (dur) => { /* 입력창 사라짐 */ },
}
```

**Step 3: 런타임 동작 (엔진의 처리)**
이제 시나리오에서 `{ type: 'input', ... }` 커맨드가 실행되면 어떤 일이 벌어질까요?

1.  **감지**: 엔진은 `input` 커맨드가 실행되면서 `input` 모듈의 뷰가 활성화됨을 인지합니다. (혹은 사용자가 우클릭으로 UI 숨기기를 시도할 때 발생합니다.)
2.  **조회**: `input` 모듈의 뷰에 적힌 `hideGroups` 목록을 확인합니다. "아, `'main-ui'`를 숨겨야 하는구나!"
3.  **실행**: 엔진은 현재 떠 있는 모든 UI 중 `uiGroup`이 `'main-ui'`인 것을 찾아, 그 모듈의 **`hide()` 메서드를 즉시 호출**합니다.
4.  **복구**: `input` 커맨드가 종료되거나 사용자가 UI를 다시 표시하면, 숨겨졌던 `'main-ui'`들은 다시 자신의 상태에 따라 필요한 시점에 `show()`가 호출되어 자연스럽게 복귀합니다.

---

#### 💡 진행 제어 (canAdvance)

`novel.next()`(클릭 또는 엔터)가 호출될 때, 엔진은 모든 UI 엔트리의 **`canAdvance`** 메서드를 확인합니다.

| 속성 | 타입 | 역할 |
| :--- | :--- | :--- |
| **`canAdvance`** | `() => boolean` | `false`를 반환하면 엔진의 진행을 막습니다. |

*   **주요 용도**: 텍스트 타이핑 중이거나 연출 중일 때 사용자의 클릭으로 대사가 넘어가는 것을 방지합니다.
*   **작동 원리**: 하나라도 `false`를 반환하면 `novel.next()`는 무시됩니다.

## 📸 렌더러 연동 규칙 (`Renderer`)

`ctx.renderer`를 통해 캔버스 자원을 조작할 때는 다음 규칙을 준수해야 합니다.

1.  **객체 추적 (`track`)**: 생성한 모든 객체는 `track(obj)`을 거쳐야 합니다. 이를 통해 씬 전환 시 메모리가 자동 해제됩니다.
2.  **시간 보정 (`dur`)**: 모든 시간값은 `dur(ms)`로 감싸야 합니다. 이는 유저의 **Skip 모드** 상태에 따라 소요 시간을 자동으로 보정합니다.
3.  **애니메이션 (`animate`)**: 내장 애니메이션 유틸리티를 사용하면 스킵 모드와 연동되어 연출 시간을 제어할 수 있습니다.
