# 02. 프로젝트 설정 (Configuration) ⚙️

모든 `fumika` 프로젝트의 중심에는 `novel.config.ts` 파일이 있습니다. 이 파일은 게임의 해상도부터 어떤 캐릭터가 등장하는지까지 모든 정보를 담고 있는 **설계도**와 같습니다.

---

## `novel.config.ts` 작성

루트 폴더에 파일을 생성하고 아래 내용을 작성합니다.

```ts
import { defineNovelConfig } from 'fumika'

export default defineNovelConfig({
  // 1. 화면 해상도 설정 (논리 좌표계)
  width: 1280,
  height: 720,

  // 2. 전역 변수 초기값
  variables: { 
    score: 0 
  },

  // 3. 사용할 씬 목록
  scenes: ['scene-intro'],

  // 4. 에셋 정의 (물리적 파일과 논리적 이름의 연결)
  assets: {
    aris_normal: './assets/aris.png',
    bg_school: './assets/school.jpg',
    sakura_petal: './assets/sakura.png' // 06단계 이펙트를 위해 추가
  },

  // 5. 배경 정의
  backgrounds: {
    school: { src: 'bg_school' }
  }
})
```

## 💡 왜 에셋(`assets`)과 배경(`backgrounds`)을 따로 적나요?

이미지 파일 경로를 곧바로 배경에 쓰지 않고, `assets`를 거쳐서 사용하는 데에는 중요한 이유가 있습니다.

**예를 들어**, `school.jpg`라는 하나의 이미지 파일을 상황에 따라 두 가지 방식으로 쓰고 싶다고 가정해 봅시다.
1.  카메라 이동에 따라 입체감이 느껴지는 배경 (`parallax: true`)
2.  단순히 화면에 꽉 차게 보여주는 일러스트 (`parallax: false`)

이때 `assets`에 파일 경로를 한 번만 등록해 두면, `backgrounds`나 다른 설정에서 이 키값(`bg_school`)을 돌려쓰며 각기 다른 속성을 부여할 수 있습니다. 즉, **리소스의 물리적 위치(파일 경로)와 논리적 역할(배경, 캐릭터 등)을 분리하여 재사용성을 높이기 위함**입니다.

## 💡 왜 전역 변수(`variables`)를 미리 선언하나요?

엔진에게 "이 게임에서 사용할 데이터는 이것들이다"라고 미리 알려주는 과정입니다. 

**예를 들어**, `score`라는 변수를 미리 등록해 두지 않고 시나리오 도중에 갑자기 사용한다면, 엔진은 이 값이 세이브 파일에 저장되어야 하는 중요한 정보인지 아니면 잠시 쓰고 버릴 정보인지 판단할 수 없습니다. 미리 선언해 둠으로써 **안정적인 세이브/로드 시스템을 보장**받게 됩니다.

## 💡 논리 좌표계 (Resolution)의 이해

`width`와 `height`는 실제 브라우저 창의 크기와는 독립적인 **논리적인 좌표**입니다. 

**예를 들어**, 설정을 `1280x720`으로 했다면, 게임 화면의 가장 정중앙 좌표는 항상 `(640, 360)`이 됩니다. 사용자가 창 크기를 늘리거나 스마트폰으로 접속해도 엔진이 자동으로 비율을 맞춰주기 때문에, 개발자는 해상도 대응에 스트레스받지 않고 중앙 좌표값만 기억하면 됩니다.
